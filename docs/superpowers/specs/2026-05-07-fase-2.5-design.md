# Fase 2.5 — Filtros estructurados + smart mock + debug tools · Design

**Fecha:** 2026-05-07
**Estado del repo al diseñar:** Fase 2 cerrada (`docs/superpowers/reports/2026-05-07-fase-2-cierre.md`).
**Branch propuesta:** `feat/fase-2.5-filters-mock-debug` (desde `feat/fase-2-hybrid-search`).

## 1. Por qué Fase 2.5 existe

Auditoría honesta del eval de Fase 2 (con criterio del negocio, no inflado): **48% de queries no-edge fallan parcialmente** (12 de 25). Causas raíz identificadas:

1. **Bug contra spec — Filtros estructurados faltantes.** Master doc Sec 9 Paso 4b dice *"cosine con filtros estructurados aplicados (categoría, género, edad, precio)"*. Fase 2 implementa SOLO `categoría`. El LLM extrae `recipient_gender`, `recipient_age_min/max`, `price_range` pero esos campos se persisten en `searches` sin nunca filtrar.

2. **Bug contra spec — Freshness por categoría faltante.** Master doc Sec 9 Paso 5 dice *"Si los resultados locales están por debajo del umbral Y la categoría no se ha refrescado recientemente Y confidence > 0.5, se llama al mock"*. Fase 2 implementa SOLO los 2 primeros criterios. En producción esto significa pagar `$0.04` cada vez que un usuario busca algo no-cacheado en una categoría que ya refrescamos hace 5 minutos.

3. **Mock no realista.** El mock fixture estático tiene 10 plantillas genéricas. Cuando el usuario busca "Nike Air Max 270", el mock devuelve random-de-categoría — no productos Nike. Esto rompe el flujo de mock fallback en escenarios realistas y degrada la calidad del eval.

4. **Cero observabilidad.** No hay forma de inspeccionar qué pensó el sistema antes de devolver resultados. Cuando una query da resultados raros, no sabes si fue el LLM normalize, BM25, cosine, RRF, o el mock fallback.

**Lo que Fase 2.5 NO incluye** (sigue diferido a Fase 3+):
- Vector de perfil del usuario, multi-modo, k-means semanal — Fase 3a/3b.
- Reranking por perfil (Paso 6 spec) — Fase 3a.
- LLM reranker contextual + razones generadas — Fase 3c.
- LangGraph para orquestación — evaluado y rechazado para Fase 2.5 (overkill para un pipeline lineal con 2 condicionales). Reconsiderar para Fase 3c si el reranker tiene grafo no-trivial.
- UI admin completo, role-based access — Fase 4.

## 2. Decisiones de scope tomadas durante el brainstorming

| Decisión | Elección | Razón |
|---|---|---|
| Scope F2.5 | Bug fixes + smart mock + debug tools (las 3 cosas) | Cierra Fase 2 honesta antes de entrar a personalización |

### 3.1 Filtros estructurados extendidos (Bug #1)

**Cambio en `src/sectors/c-search/retrieve/bm25.ts` y `cosine.ts`:**

```ts

  categories?: string[];
  gender_target?: 'femenino' | 'masculino' | 'unisex';
  age_min?: number;
  age_max?: number;
  price_range?: 'bajo' | 'medio' | 'alto';

```

**Mapping `price_range` → `price_cents` ranges:**
- `bajo` → `[0, 1999]` cents (≤ $19.99)
- `medio` → `[2000, 9999]` cents ($20 - $99.99)
- `alto` → `[10000, ∞)` cents (≥ $100)

**SQL aplicado en BM25 y cosine** (mismo predicado en ambas funciones):

```sql

  -- BM25-only: AND tsvector_es @@ websearch_to_tsquery('spanish', $1)
  -- cosine-only: AND embedding IS NOT NULL
  AND ($cats::text[] IS NULL OR (metadata->>'category') = ANY($cats::text[]))
  AND (
    $gender::text IS NULL
    OR (metadata->>'gender_target') IS NULL
    OR (metadata->>'gender_target') = 'unisex'
    OR (metadata->>'gender_target') = $gender
  )
  AND (
    $age_min::int IS NULL
    OR (metadata->'age_target'->>'max') IS NULL
    OR (
      COALESCE((metadata->'age_target'->>'max')::int, 999) >= $age_min
      AND COALESCE((metadata->'age_target'->>'min')::int, 0) <= $age_max
    )
  )
  AND (
    $price_range::text IS NULL
    OR ($price_range = 'bajo'  AND price_cents BETWEEN 0 AND 1999)
    OR ($price_range = 'medio' AND price_cents BETWEEN 2000 AND 9999)
    OR ($price_range = 'alto'  AND price_cents >= 10000)
  )
```

### 3.2 Freshness por categoría en `shouldCallMock` (Bug #2)

**Nuevo helper:**

```ts
// src/sectors/c-search/decide/freshness.ts (NEW)
export async function getCategoryFreshness(
  category: string | null,
  pg: Client,
): Promise<Date | null> {
  if (!category) return null;
  const r = await pg.query(
    `SELECT MAX(last_refreshed_at) AS last_refresh
     FROM products
     WHERE metadata->>'category' = $1 AND is_active = true`,
    [category],
  );
  const t = r.rows[0]?.last_refresh;
  return t ? new Date(t) : null;
}
```

**Cambio en `decide/shouldCallMock.ts`:**

```ts
export const LOCAL_HITS_THRESHOLD = 12;
export const CONFIDENCE_THRESHOLD = 0.5;
export const FRESHNESS_THRESHOLD_HOURS = 24;

export function shouldCallMock(
  localCount: number,
  confidence: number,
  lastRefreshedAt: Date | null,
): boolean {
  if (localCount >= LOCAL_HITS_THRESHOLD) return false;
  if (confidence <= CONFIDENCE_THRESHOLD) return false;
  if (lastRefreshedAt) {
    const ageHours = (Date.now() - lastRefreshedAt.getTime()) / (3600 * 1000);
    if (ageHours < FRESHNESS_THRESHOLD_HOURS) return false;
  }
  return true;
}
```

**En `search.ts` orchestrator:**

```ts

  normalized?.categories?.[0] ?? null,
  pg,
);
if (normalized && shouldCallMock(fused.length, normalized.confidence, lastRefreshedAt)) {
  // ... mock fallback
}
```

**Tests** (extender `decide-mock.test.ts` + nuevo `freshness.test.ts`):
- `shouldCallMock returns false if category refreshed < 24h ago even with low count + high confidence`
- `shouldCallMock returns true if last refresh > 24h ago and other criteria met`
- `shouldCallMock returns true if no last refresh (no products in category)` — current behavior preserved
- `getCategoryFreshness returns MAX(last_refreshed_at) for matching products` — integration test
- Integration test del orchestrator: refrescar productos via mock → segunda query inmediata con count bajo NO dispara segundo mock call.

## 4. Bloque 2 — Smart mock LLM-generated

### 4.1 Estrategia: mode switch (no breaking change)

**Modos** (env var `MOCK_AGGREGATOR_MODE`, default `auto`):
- `auto` — usa LLM si la llamada incluye `query` no-vacía; usa fixture en otro caso (cron, healthchecks).
- `fixture` — siempre fixture (default explícito en tests deterministas).
- `llm` — siempre LLM (modo agresivo de realismo).
- `fixture-only` — siempre fixture, retorna `[]` si no matchea (legacy comportamiento Phase 0).

**Por qué este modo split:**
- Phase 0 `tests/integration/mock-aggregator.test.ts` valida distribución 40/20/15/10/10/5%, latencia 2-4s, error rate 2% → tests no llaman con `query` → modo `auto` cae en `fixture` → tests siguen verdes.
- Cron `runCatalogFill` llama por categoría sin `query` específico → fixture (mantiene determinismo del cron).
- Search mock fallback llama con `query: normalized.search_terms` → LLM genera realista.

- price_cents: realista para la categoría (electrónica $5000-$200000, ropa $500-$20000, etc.).
- title: corto y descriptivo (max 80 chars).
- description: 2-3 frases (max 200 chars).
- image_url: "https://placehold.co/400x400.png".
- attributes: objeto con propiedades específicas (color, size, material, capacity, age_target, etc.).

Devuelve SOLO un objeto JSON con shape: { "products": [...] } donde "products" es un array de N productos. Sin markdown wrap, sin texto adicional.`;

const productSchema = z.object({
  source: z.enum(["amazon", "aliexpress", "shein"]),
  source_product_id: z.string().min(1).max(64),
  title: z.string().min(1).max(120),
  description: z.string().max(400),
  image_url: z.string(),
  price_cents: z.number().int().min(0).max(9999999),
  brand: z.string().max(80),
  raw_category: z.string().min(1),
  attributes: z.record(z.string(), z.unknown()),
});

const responseSchema = z.object({
  products: z.array(productSchema).min(1).max(50),
});

export interface GenerateOpts {
  query: string;
  category?: MockCategory;
  limit: number;
}

export async function generateProductsWithLLM(opts: GenerateOpts): Promise<MockProduct[]> {
  const userMsg = JSON.stringify({
    query: opts.query,
    category: opts.category ?? "any",
    limit: opts.limit,
  });
  const res = await defaultProvider.chat({
    system: SYSTEM_PROMPT,
    messages: [{ role: "user", content: userMsg }],
    maxTokens: Math.min(4000, 200 * opts.limit),  // ~200 tokens por producto
    temperature: 0.7,
    jsonMode: true,
    cacheSystem: false,  // queries únicas, no caching
  });
  const text = stripMarkdownWrapper(res.text);
  const parsed = JSON.parse(text);
  const valid = responseSchema.parse(parsed);
  return valid.products.slice(0, opts.limit).map((p) => ({
    id: randomUUID(),
    source: p.source,
    source_product_id: p.source_product_id,
    title: p.title,
    description: p.description,
    image_url: p.image_url || "https://placehold.co/400x400.png",
    price_cents: p.price_cents,
    brand: p.brand,
    raw_category: p.raw_category,
    attributes: p.attributes,
  }));
}

  callCount++;

### 5.2 Tracer helper (`src/sectors/c-search/debug/tracer.ts` NEW)

```ts
export interface ITracer {
  start(label: string): void;
  end(label: string): void;
  set<K extends keyof SearchTrace>(key: K, value: SearchTrace[K]): void;
  finish(): SearchTrace;
}

export class Tracer implements ITracer {
  private trace: Partial<SearchTrace>;
  private timers = new Map<string, number>();
  private startTime: number;
  constructor(rawQuery: string) {
    this.startTime = performance.now();
    this.trace = {
      raw_query: rawQuery,
      retrieval: { bm25: [], cosine: [], fused: [] },
      cache: { exact_hit: false, semantic_hit: false },
      filters_applied: {},
      freshness: { category_checked: null, last_refreshed_at: null, hours_old: null },
      decision: { should_call_mock: false, reason: "not evaluated" },
      mock_fallback: { invoked: false },
      timings_ms: {} as SearchTrace["timings_ms"],
    };
  }
  start(label: string) { this.timers.set(label, performance.now()); }
  end(label: string) {
    const s = this.timers.get(label);
    if (s !== undefined) {
      (this.trace.timings_ms as Record<string, number>)[label] = performance.now() - s;
    }
  }
  set<K extends keyof SearchTrace>(key: K, value: SearchTrace[K]) { (this.trace as Record<string, unknown>)[key] = value; }
  finish(): SearchTrace {
    (this.trace.timings_ms as Record<string, number>).total = performance.now() - this.startTime;
    return this.trace as SearchTrace;
  }
}

export class NoopTracer implements ITracer {
  start() {}
  end() {}
  set() {}
  finish(): SearchTrace { throw new Error("NoopTracer.finish() should never be called"); }
}
```

## 11. Riesgos

1. **Smart mock costs en tests**: ~$0.005-0.01 por full integration suite. Acotado, OK.
2. **DeepSeek puede generar productos inventados que no existirían**: por diseño — es un mock, no un agregador real. En producción se reemplaza con AliExpress API real.
3. **Trace mode aumenta tamaño de respuesta**: ~5-10 KB por query con trace. Solo en debug=true. Negligible.
4. **Fallback a fixture si LLM falla**: garantizado por try/catch. Si DeepSeek está caído, search sigue funcionando con fixture.
5. **Eval después de F2.5**: si BIEN no sube a ≥72%, los filtros no están funcionando como espero. Debug page lo dirá.
